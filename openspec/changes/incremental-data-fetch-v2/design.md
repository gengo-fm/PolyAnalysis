## Context

### 当前状态

- **数据获取**：每次运行都从 Polymarket API 拉取全量数据
- **缓存**：存在但未被有效利用
- **进度**：无持久化，中断后丢失

### 问题

1. **效率低**：1300+ 钱包需要 10+ 小时
2. **重复获取**：已有缓存仍重新请求 API
3. **不可恢复**：中断需重新开始

### 约束

- Polymarket API 限流：60 请求/分钟
- 文件系统存储
- Python 异步编程

## Goals / Non-Goals

**Goals:**
- 实现三层缓存架构（内存 → 文件 → API）
- 支持基于时间戳的增量更新
- 实现断点续传
- 提升整体处理速度 10x+

**Non-Goals:**
- 不修改现有 API 响应格式
- 不实现分布式处理
- 不支持多钱包并行分析（单进程内并发）

## Decisions

### 决策 1: 三层缓存 vs 单层缓存

**选择**: 三层缓存（内存 + 文件 + API）

**理由**: 
- 内存缓存最快，适合同一会话内多次查询
- 文件缓存持久化，支持中断恢复
- API 作为最终数据源

**备选**: 仅文件缓存
- 缺点：每次都读磁盘，速度慢

### 决策 2: 时间戳增量 vs 指纹检测

**选择**: 基于时间戳的增量

**理由**:
- Polymarket API 原生支持 `since` 参数
- 实现简单，可靠性高

**备选**: 数据指纹（hash）
- 缺点：需要全量对比，API 无此功能

### 决策 3: 异步并发方案

**选择**: asyncio + Semaphore

**理由**:
- 原生 Python 支持
- 配合限流实现简单

**备选**: 多进程
- 缺点：复杂度高，资源共享难

### 决策 4: 进度存储

**选择**: 独立 `_progress.json` 文件

**理由**:
- 结构清晰，易于调试
- 支持断点恢复

## Risks / Trade-offs

### Risk 1: API 限流影响并发

**→ Mitigation**: 
- 实现请求队列
- 超过限制自动等待
- 失败自动重试

### Risk 2: 缓存数据过期

**→ Mitigation**:
- 24 小时后自动刷新
- 用户可强制刷新

### Risk 3: 磁盘空间

**→ Mitigation**:
- 每个钱包约 1-10MB
- 1000 钱包约 1-10GB
- 定期清理 30 天前数据

### Risk 4: 数据一致性

**→ Mitigation**:
- 原子写入
- 损坏文件自动跳过
